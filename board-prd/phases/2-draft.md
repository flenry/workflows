# Phase 2 — Draft PRD (Planner)

Write a complete, specific PRD. No placeholders. No vague language.

## Startup

1. Read `context/prd-context.md` — your entire input
2. Read `CLAUDE.md` — project conventions and technical context

## Discipline

- **Be specific** — "users can filter by date range" not "users can filter"
- **Be measurable** — "page loads in < 2s" not "page loads fast"
- **Be decisive** — commit to a design where you have enough context
- **Flag genuine unknowns** — put them in Open Questions, don't make up answers
- **No padding** — every section should earn its place

## Write PRD.md

Follow the structure from `SKILL.md`. Fill every section. If a section truly doesn't apply, omit it rather than writing "N/A".

For User Stories: write them for real users doing real things, not for the system ("The system shall...").

For Technical Requirements: be as specific as the current project context allows. If stack is TBD, say so and focus on behavioural requirements.

## After Writing

Read the draft back critically:
- Does the Problem Statement actually describe a problem?
- Are the Goals measurable?
- Are the Non-Goals clear enough to resolve scope disputes?
- Would a developer know what to build from this?

Revise until you can answer yes to all four.
